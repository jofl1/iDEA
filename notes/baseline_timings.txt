Summary
==============================================================================

| Case               | Phase A wall | Phase B wall | Speedup | Phase A mem | Phase B mem | Mem reduction |
| ------------------ | -----------: | -----------: | ------: | ----------: | ----------: | ------------: |
| interacting_uu_80  |       0.166s |       0.122s |   1.36x |     129 MB  |     100 MB  |          -23% |
| interacting_uud_60 |      26.146s |       1.062s |  24.62x |    1250 MB  |     160 MB  |          -87% |
| interacting_uudd_30|     (fails)  |       1.666s |     N/A |    (failed) |     244 MB  |           N/A |

Energy and density match between solvers to within 1e-13 across all cases
(checked via tests/test_interacting_det.py with rtol=1e-10).

The uu_80 case sees a 2x dimension cut from same-spin Pauli (labelled
80^2=6400 -> determinant C(80,2)*C(80,0)=3160 since `electrons="uu"`
means 2 up + 0 down, not 1 up + 1 down). Combined with skipping
antisymmetrize() and avoiding the labelled _estimate_level=18
overcompute ((|2-0|+1)^2 * 2 * 1 = 18 eigenpairs vs 1 in determinant),
the modest 1.36x gain is what we'd expect at this tiny problem size
where Python/ARPACK setup dominates.

The uud_60 case has a ~2x raw dimension cut (labelled 60^3=216,000 ->
determinant C(60,2)*C(60,1)=106,200). The 25x wall-time speedup comes
mostly from elsewhere: avoiding _estimate_level=12 overcompute,
avoiding the labelled CSR Hamiltonian construction, and avoiding the
post-solve antisymmetrize+filter+deduplicate pass. The ~8x memory
reduction is dominated by skipping the global Kronecker CSR.

The uudd_30 case is the qualitative win: the labelled solver can't reliably
produce this case at all, and the determinant solver does it in under 2
seconds with stable runs.

==============================================================================
Phase C+ — determinant basis + parity fast path (single predicted block)
==============================================================================

Same matrix-free LinearOperator as Phase B but the solver now exploits
spatial parity symmetry. Only the predicted ground-parity block is
solved (the friend's formula
p_gs = (-1)^(N_up*(N_up-1)/2 + N_down*(N_down-1)/2)),
via the implicit-projector form eigsh(Q_p^T H Q_p).

The implicit-projector matvec still scatters into the full basis on
each step, so the speedup is bounded below the asymptotic 2x. The
friend's prediction: "if the measured speedup is < 1.3x, that's the
signal to schedule the compressed-basis follow-up (Phase C++) before
moving to Phase D". We're right at the threshold.

==============================================================================
Phase D — consolidation: transparent dispatch from interacting.solve
==============================================================================

After Phase D, ``iDEA.methods.interacting.solve`` transparently dispatches
to ``iDEA.methods.interacting_det.solve`` whenever conditions allow
(ground state ``k=0``, default Hamiltonian, CPU, electrons string contains
only 'u'/'d'). Existing scripts get the speedup automatically — no import
changes, no API breakage.

The returned ``ManyBodyState`` has ``.energy``, ``.space``, ``.spin``, and
``.full`` all populated, so observables built on ``.full`` (e.g.
``iDEA.observables.density_matrix``) continue to work unchanged.
``build_labelled_state`` reconstructs ``.full`` via the existing
``antisymmetrize`` call and derives ``.space`` from the canonical spin
slice with the spin-multiplicity correction
(``space = sqrt(N_e! / (N_up! * N_down!)) * full[canonical slice]``).

Escape hatch: ``bypass_det=True`` keyword forces the legacy labelled
solver for debugging or unusual configurations.

Re-measure the dispatched path with:

The dispatched-via-labelled path is slightly slower than calling
``interacting_det.solve`` directly because it incurs the
``antisymmetrize`` reconstruction overhead so that ``.full`` is
available. This overhead is bounded (O(N^Ne * Ne!)) and the determinant
solve still dominates wall time. Users who only need energy and density
can call ``interacting_det.solve`` directly to skip the reconstruction.

==============================================================================
Phase E — PRIMME backend + cold-start v0 (Phase E2 preconditioner deferred)
==============================================================================

After Phase E, ``iDEA.methods.interacting_det.solve`` routes both
eigsh call sites through a backend dispatch (PRIMME if installed and
operator dim >= ``IDEA_DET_PRIMME_MIN_DIM``, default 2000; scipy
otherwise). PRIMME's initial guess is a non-interacting Slater
determinant in the grid-determinant basis: build the single-particle
Hamiltonian ``K + diag(v_ext)``, diagonalise, expand the lowest
``N_up`` and ``N_down`` orbitals as Slater minors per channel, outer-
product.

PRIMME is an optional dependency declared under the new
``[project.optional-dependencies.fast]`` group; install via
``pip install -e '.[fast]'``. When absent, the dispatch falls back to
scipy with no functional change.

A new ``--count-matvecs`` flag on ``benchmark_interacting.py time``
instruments matvec counts via ``benchmarks/counting_operator.py``
(friend's red flag #7: "Do not benchmark only wall time").

Phase E2 (diagonal Jacobi preconditioner) was attempted and skipped.
Empirically the eigenvalue is ~60 below ``min(diag(H))`` on these
cases (kinetic energy dominates the diagonal in the grid basis), so
``(diag - shift)`` is nearly uniform across basis states and the
preconditioner is essentially identity. Friend's advice doc
anticipated this: the orbital-basis preconditioner (H_0 diagonal in
single-particle orbitals) is the right next step but needs a basis
transform. Deferred to a future phase.

2.0e-13) over its noise floor. See the long-suite section below.
